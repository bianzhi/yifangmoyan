# 经验总结文档

> 记录开发过程中的关键经验教训，避免重复犯错。

---

## 一、一切基于定义出发

### 教训来源
缠论笔模块（bi.rs）间隔检查的 bug，导致 002015 等股票笔划分错误，反复修复花费大量时间。

### 问题描述
缠论定义新笔要求"顶底分型之间至少有1根独立K线"。实现时将定义"等价转换"为**两个分型中间K线（k2）的索引差≥3**，看似正确实则不等价：

```
顶分型 bars=[5,6,7]  底分型 bars=[8,9,10]
  k2索引=6              k2索引=9
  9 - 6 = 3 ≥ 3 → 通过 ✗（实际无独立K线！）

正确：底分型第一根 - 顶分型最后一根 = 8 - 7 = 1 < 2 → 不通过 ✓
```

### 核心原则

> **先翻译定义，再优化实现。定义说什么就写什么，先写最直白的代码。等价转换只有在严格证明后才可使用。**

### 实践方法
1. **直译定义**：把数学定义/业务规则逐字翻译为代码条件，不做"聪明"的转换
2. **边界验证**：每个条件都测试"刚好满足"和"刚好不满足"两种边界
3. **举反例**：写完条件后，构造数据验证"看似等价但不等价"的反例场景

## 二、上游优先：先修病因，再修症状

### 教训来源
笔模块间隔检查错误 → 产生不应该存在的笔 → 笔端点不连续 → 线段方向错误

### 问题描述
笔的间隔检查有 bug，导致产出了不该存在的笔对。这些错误数据传递到下游，表现为：
- 笔端点价格不连续（相邻笔共享端点但价格对不上）
- 线段方向与价格矛盾（上升线段但 start ≥ end）
- 匹配算法需要各种补丁来兼容错误数据

我在下游花了 ~9 轮修复这些症状，而不是回溯到最根本的"笔是如何产生的"这一步骤。每修一个下游症状，新的下游问题又暴露出来，因为上游在持续产生错误数据。

### 核心原则

> **下游出错时，先验证上游数据是否正确，而不是急着修下游。上游数据正确，下游问题自然消失。**

### 实践方法
1. **从源头验证**：发现下游 bug 时，先检查其输入数据是否正确
2. **逐层排查**：数据流 → 模块A输出 → 模块B输入 → 模块B输出 → ... 定位第一个出错的环节
3. **不要补丁**：如果上游数据错误，修上游；不要在下游加兼容逻辑掩盖问题

## 三、测试覆盖要包含边界场景

### 教训来源
间隔检查的 bug 没被测试捕获，因为测试数据的 K 线恰好"理想"——bar_index 差值≥3 时也确实有独立 K 线。

### 问题描述
测试中构造的 K 线数据，分型之间总是有明显的间隔，没有覆盖"bar_index 差值满足但实际无独立 K 线"的边界场景。导致 bug 长期潜伏，直到用真实股票数据（002015）才暴露。

### 核心原则

> **测试不仅要验证正常场景，更要覆盖定义的边界——特别是你做了"等价转换"的地方。**

### 实践方法
1. **边界优先**：每个条件写出 `n` 和 `n-1` 两个测试用例
2. **反例测试**：对"等价转换"或"简化"的逻辑，专门构造理论上不满足原始定义但满足转换后条件的反例
3. **真实数据验证**：模块完成后，立即用真实市场数据端到端验证，不要只依赖手工构造的理想数据

## 四、线段重构的耻辱：改后端却炸了前端K线

### 教训来源
线段模块（xd.rs）重构期间，前后端交互的 bug 导致K线图反复不显示。从 `4642508` 重构线段开始到 `9169094` 最终修复，中间产生了 **7次** 专门修复"K线不显示"的提交，横跨20多个版本。

### 完整翻车时间线

| # | 提交 | 说法 | 根因 |
|---|------|------|------|
| 1 | `fdd9807` | 覆盖层异常隔离+重复调用删除 | 缠论覆盖层抛异常导致整个图表崩溃 |
| 2 | `2f80475` | 覆盖层使用正确的klineOffset偏移 | 覆盖层计算时间偏移错误 |
| 3 | `881d353` | K线蜡烛不可见（光标/价格正常） | 图表尺寸/布局问题——autoSize未启用 |
| 4 | `7f42617` | K线不显示和日志区不可复制 | 多个UI问题叠加 |
| 5 | `40f8921` | 前端TS编译错误导致图表无法构建 | 后端数据结构变了，前端没跟上 |
| 6 | `dfcdaad` | 勾选线段后K线不显示 | 线段渲染逻辑与笔冲突（priceScaleId、数据覆盖） |
| 7 | `9169094` | K线图不显示问题最终修复 | 清理了全部调试代码和残留兼容逻辑 |

7次前端紧急修复。每次以为修好了，换另一个场景又白屏。后端改线段算法，用户看到的是K线消失了。

### 根因分析：为什么改后端会炸前端？

**本质错误：后端返给前端的数据格式/内容不稳定，前端没有防御性处理。**

1. **线段数据含 NaN/0/Infinity 异常值**：线段算法重构期间，`start_price`/`end_price` 可能返回0或NaN。LightweightCharts 收到非法数据直接崩溃，整个图表白屏。前端从未校验过数据有效性。

2. **索引越界**：线段的 `start_index`/`end_index` 超出 klines 数组范围。后端认为第N根K线存在，前端 klines 只有M根（M<N）。直接 `data.klines[xd.start_index]` 得到 undefined，后续 `.dt` 爆了。

3. **时间序违反**：线段合并为单条 LineSeries 时，时间序必须严格递增。但线段共享端点导致同一时间点出现两条线段，合并后时间序被破坏，LightweightCharts 拒绝渲染整个 series。

4. **后端反复改动数据格式，前端追踪不紧**：后端重构线段时改了字段名、改了价格计算方式、改了索引含义，但 `invoke('get_chart_data')` 的 TS 类型定义没有同步更新。编译过了≠运行时正确。

### 更深层的错误

1. **没有前后端契约**：Rust 后端和 Vue 前端之间只有一个 `invoke` 调用，没有 schema 校验。后端随便改返回结构，前端只能在运行时崩溃才发现。

2. **覆盖层异常没隔离**：缠论覆盖层（笔、线段、分型）的渲染异常会让整个图表白屏。一个 `bi.start_price = 0` 就能炸掉整个 K 线图。直到 `fdd9807` 才加了 try-catch，但这是第5次K线不显示之后了。

3. **没有增量验证**：重构线段时一口气改完再测试。如果每次改后端都同步验证前端显示，第一次就能发现K线消失了，而不是累积了7个不同的前端正策问题。

### 核心原则

> **后端改动必须立即验证前端显示。前后端没有类型契约的项目，任何后端数据变更都是定时炸弹。**

### 实践方法
1. **前端防御性编程**：图表渲染必须 try-catch 隔离每个覆盖层；数据使用前必须校验（NaN/0/Infinity/越界），不能信任后端永远返回合法数据
2. **前后端类型同步**：Rust struct 改了字段，TypeScript 接口必须同步改。最好用 tauri 的 command 宏自动生成 TS 类型，不要手写两遍
3. **后端改了就验证前端**：CI 或手动流程中，每次后端数据结构变更后必须在前端打开看一眼，不要"后端测试通过就行"
4. **共享端点的时间序**：线段画图时，不能把多条线段合并为一条 LineSeries（时间会重复）。每条线段用独立 LineSeries

## 五、缠论实现的关键经验

### 5.1 分型（Fenxing）

- **去包含优先**：识别分型前必须先做 K 线包含处理，否则包含关系会破坏分型判断
- **交替性**：顶底分型必须交替出现。当连续出现同类型分型时，保留极值更极端的那个（顶取更高，底取更低）
- **方向约束**：配对时顶的 fx 必须高于底的 fx，否则跳过

### 5.2 笔（Bi）

- **四步流程**：识别分型 → 配对顶底 → 检查失效 → 创建笔，每步职责清晰
- **间隔检查**：用 `bars` 实际覆盖范围而非 `bar_index` 中心点差值。`next.bars[0] - curr.bars[2] >= 2` 才是正确翻译
- **失效锁定**：反向分型的 k1 出现即锁定前分型，后续新高/新低不影响
- **端点连续**：相邻笔共享分型端点，第 N 笔的 end 分型 = 第 N+1 笔的 start 分型
- **价格取值**：笔的起止价格根据方向取分型的高/低点：
  - 上升笔（底→顶）：start_price = 底分型低点, end_price = 顶分型高点
  - 下降笔（顶→底）：start_price = 顶分型高点, end_price = 底分型低点

### 5.3 线段（XianDuan）

- **共享端点**：相邻线段共享端点笔，该笔是前一线段的 end_bi 也是后一线段的 start_bi
- **方向判定**：线段方向由第一笔方向决定，不是前一线段方向的取反
- **价格取值**：共享端点场景下，线段的 start_price 需根据线段方向取该笔的 high 或 low

## 六、通用开发经验

### 6.1 调试策略
- **二分法定位**：bug 出现在第 N 个输出但不在第 N-1 个，问题一定在两者之间的处理逻辑
- **打印中间结果**：复杂算法出错时，打印每一步的中间结果（分型列表、配对结果、失效检查），比盲猜高效
- **最小复现**：用最少的 K 线数据复现问题，减少干扰因素

### 6.2 代码组织
- **职责单一**：每个函数只做一件事。分型识别、配对、失效检查、笔创建各是一个函数，不要混在一起
- **数据结构先行**：先定义清晰的数据结构（FxResult、FxPair、Bi），再写处理逻辑
- **避免补丁累加**：如果代码需要越来越多的补丁来兼容各种场景，说明核心逻辑有误，应重新审视而非继续打补丁

### 6.3 Rust 特有
- **所有权设计**：函数签名尽量用引用（`&[T]`）而非所有权转移，减少不必要的 clone
- **测试隔离**：每个测试函数用独立的数据构造，不要依赖其他测试的副作用
- **编译器是朋友**：警告（unused variable 等）也要处理，它们往往指向逻辑问题
